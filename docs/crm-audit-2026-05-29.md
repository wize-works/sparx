## Fail list

### F-01 — Module gate is inconsistent across CRM surfaces — **this is the #1 fix**

- **Severity:** critical / **blocker**
- **Route(s) affected:**
  - `/crm` — renders ModuleStub when CRM is Active (should render the customer list)
  - `/crm/duplicates` — server-redirects to `/crm` → falls into the same ModuleStub
  - customer-create Server Action — rejects with `Module "crm" is not active for this tenant` immediately after a successful Activate click on `/settings/modules`
  - `/crm/pipelines`, `/crm/segments`, `/crm/tasks`, `/crm/b2b`, `/crm/reports`, `/crm/orders`, `/crm/quotes` — render full CRM UI **even when `/settings/modules` reports all 8 modules Inactive** (i.e., gate is _not_ enforced on the sub-routes)
- **Repro:**
  1. Sign in to `https://app.sparx.works` as `e2e-staff@sparx.test`.
  2. Visit `/settings/modules`. All 8 modules show **Inactive** status pills, each with an `Activate` button.
  3. Click `Activate` on CRM. Status pill flips to **Active**; button becomes `Deactivate`. Bootstrap fires — `/crm/pipelines` lists the seeded Sales Pipeline with 6 stages, `/crm/segments` lists the 4 built-ins.
  4. Navigate to `/crm`. The page renders the ModuleStub ("CRM is coming online", "Planned" feature cards) — _not_ the customer list. Hard reload + `Cache-Control: no-cache` — same result.
  5. Navigate to `/crm/customers/new`. The form renders. Fill First name + a deliberate bad email. Click `Create customer`. HTML5 client validation fires on the email field; the form still POSTs to the server action; the server returns role=alert: `Module "crm" is not active for this tenant`.
  6. Navigate back to `/settings/modules`. CRM has reverted to **Inactive** (all 8 modules now show Inactive again — even though I never deactivated). Hard reload confirms.
  7. Navigate to `/crm/pipelines` — still renders the Sales Pipeline card with full CRM UI despite the gate now reporting Inactive.
- **Screenshot:** [docs/crm-audit/F-02-crm-root-modulestub-when-active.png](crm-audit/F-02-crm-root-modulestub-when-active.png) (captured under the old F-02 ID before re-numbering; same evidence — `/crm` showing ModuleStub while CRM was Active in `/settings/modules`).
- **Suspected cause:** at least two independent module-state sources in the dashboard, and the write-side Server Action source disagrees with the read-side Server Component source:
  - The Server Action at `customers/new`'s submit handler reads the gate and sees Inactive.
  - The Server Component at `/crm/page.tsx` also reads from a source that doesn't see CRM as Active immediately after activation.
  - The Server Components at `/crm/pipelines/page.tsx`, `/crm/segments/page.tsx`, etc. don't consult the module gate at all (or consult one that always says Active for tenants that have ever had the seeds).
  - The `/settings/modules` page's status pill reads yet another source that drops back to Inactive on the next page load.
    Likely candidates: (a) the dashboard pod's in-process LRU module-gate cache (CLAUDE.md mentions 60s TTL; that didn't help — observed 2+ min after activation), (b) Next.js Route Cache / Data Cache not invalidated when `module.activated` fires, (c) the activation Server Action returning success without actually writing to `tenant_modules` (or writing with a TTL).
- **Suggested fix:**
  1. **Audit what each of these surfaces calls to determine "is module X active for tenant T":** `/crm/page.tsx`, `/crm/pipelines/page.tsx`, `/crm/customers/new`'s create Server Action, `/settings/modules/page.tsx`, and the Activate/Deactivate Server Action. There should be exactly one function. If there are multiple, consolidate. If there is one, audit whether each call site bypasses or short-circuits it.
  2. **Confirm `tenant_modules` is actually written.** Watch the row before/after an Activate click. If the row updates and reverts later, find the process reverting it. If the row never updates, the Server Action is lying about success.
  3. **Invalidate Next.js caches on activation.** The `module.activated` consumer should call `revalidatePath('/crm', 'layout')` (and equivalent for every gated module) so server-rendered route segments pick up the new state on the next request.
  4. **Decide gate enforcement on `/crm/(sub-routes)`.** Right now most sub-routes happily render even when the gate reports Inactive. Per CLAUDE.md the contract is "Disabled modules return 404 from their API surface, run no consumers, and store no rows" — the dashboard surface should match: an inactive module's sub-route must hit the ModuleStub (or 404), not query and render `pipelines` / `segments`.
- **Audit impact:** this blocks every mutation flow in the prompt — create customer, edit note (append-only), drag deal, Credit Hold toggle, segment save+materialize, API-key issue/revoke, module deactivate/reactivate, /welcome onboarding follow-through. It also blocks every non-browser surface (REST `/v1/crm/*`, GraphQL, MCP, webhooks, OpenAPI) because they all require an API key issued via `/settings/ai-integrations`, and key issuance is a write Server Action subject to the same gate.

### F-02 — `--module-active` resolves to Sparx Indigo `#6366f1` on every CRM page, not Cyan `#06B6D4`

- **Severity:** high (brand regression visible to every merchant who turns on CRM)
- **Route(s):** every `/crm/*` page tested — `/crm`, `/crm/pipelines`, `/crm/pipelines/[id]`, `/crm/pipelines/[id]/edit`, `/crm/segments`, `/crm/segments/[id]`, `/crm/tasks`, `/crm/b2b`, `/crm/reports`, `/crm/orders`, `/crm/quotes`, `/crm/customers/new`
- **Repro:** on any `/crm/*` page run `getComputedStyle(document.documentElement).getPropertyValue('--module-active').trim()` in the console. Result: `#6366f1`. Per `docs/sparx-brand-guide.md` + `CLAUDE.md`, CRM color is Cyan `#06B6D4`.
- **Suspected cause:** the `<ModuleProvider module="crm">` wrapper that should set `--module-active` per-route is missing from `apps/dashboard/app/(dashboard)/crm/layout.tsx`, OR the module→color token map doesn't include the `crm` entry and falls through to the default Storefront indigo, OR the layout wraps the routes but ModuleProvider's color resolver has a bug.
- **Suggested fix:** check the CRM route group layout includes `<ModuleProvider module="crm">`; verify `packages/ui/src/providers/module-provider.tsx` includes `crm → #06B6D4` in its color map (and that the `tokens.css` `--module-crm` token is defined).

### F-03 — `/crm/customers/new` has zero required-field markers

- **Severity:** medium (accessibility + UX violation per audit standards)
- **Route:** `/crm/customers/new`
- **Repro:** inspect every form input on `/crm/customers/new`. `input#firstName`, `input#lastName`, `input#email`, `input#phone`, `input#company`, `input#jobTitle`, `textarea#tags`, `input#doNotContact`, `select#type` all have `required="false"` and `aria-required=null`. The labels carry no `*` or "Required" text. The audit standard says "Form fields show `*` or 'Required' on required ones; validation errors appear inline next to the offending field, not only at top of form."
- **Suspected cause:** the form component renders all `<Label>` elements without the `required` prop, and the underlying inputs don't propagate Zod schema `required: true` to the DOM.
- **Suggested fix:** drive the Label `required` prop from the Zod schema's `optional()` shape; add `aria-required="true"` on the inputs themselves. Audit standard wants inline (not top-of-form) validation errors too — verify on the next pass once F-01 is unblocked.
